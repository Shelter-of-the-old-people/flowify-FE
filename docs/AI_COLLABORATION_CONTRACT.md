## 1. 문서 목적

이 문서는 "AI를 써서 빨리 구현한다"보다 **같은 기준으로 분석하고, 같은 기준으로 설계하고, 같은 기준으로 검증한다**를 보장하기 위해 존재한다.

이번 프로젝트에서 AI 협업의 핵심 원칙은 아래와 같다.

- 바로 코드를 쓰지 않는다.
- 현재 구현을 먼저 읽는다.
- 설계를 먼저 확인하거나 작성한다.
- 작은 단계로 구현한다.
- 구현 후에는 설계 대비 검토와 객관적 재검토를 모두 한다.

## 2. 작업 시작 전에 반드시 할 것

### 2.1 이슈 목표를 한 문장으로 고정

작업자는 먼저 아래 질문에 답해야 한다.

- 이번 이슈가 끝나면 무엇이 달라져야 하는가?
- 무엇은 이번 이슈 범위가 아닌가?

목표가 한 문장으로 정리되지 않으면 구현을 시작하지 않는다.

### 2.2 현재 구현 상태 분석

구현 전에 반드시 관련 코드를 먼저 읽는다.

- 관련 page / section / hook / API / entity / store
- 이미 구현된 것
- 비어 있는 것
- 설계와 다른 것

원칙:

- 이미 있는 구조를 모르고 새 구조를 만들지 않는다.
- "이 정도면 없겠지"라는 가정으로 구현하지 않는다.

### 2.3 설계 문서와 컨벤션 문서 확인

관련 설계 문서가 있으면 반드시 먼저 읽는다.

- 설계 문서가 있으면 현재 코드와의 차이를 먼저 정리한다.
- 설계 문서가 없으면 최소 설계를 먼저 작성한 뒤 구현한다.
- 컨벤션과 충돌하는 구현은 바로 진행하지 않는다.

### 2.4 작은 단계로 계획 수립

구현 전에 작업을 작은 단계로 쪼갠다.

- 한 번에 큰 작업을 하지 않는다.
- 파일 하나 단위로 쪼개지는 않지만, 각 단계가 독립적으로 검증 가능한 책임 단위여야 한다.
- 각 단계는 중간 상태에서도 깨지지 않아야 한다.

### 2.5 영향 범위 확인

작업 시작 전에 아래를 먼저 확인한다.

- 다른 팀원의 소유 범위와 충돌하는가?
- 공통 API path를 바꾸는가?
- response shape를 바꾸는가?
- store shape를 바꾸는가?
- panel props, route path 같은 공통 계약을 바꾸는가?

이 중 하나라도 해당되면, 구현 전에 영향 범위를 먼저 정리한다.

## 3. 작업 중에 반드시 지킬 것

### 3.1 현재 이슈 범위를 벗어난 주제를 꺼내지 않는다

작업 중 다른 문제를 발견해도 바로 손대지 않는다.

- 별도 메모
- 후속 이슈
- 설계 문서 TODO

중 하나로 남기고 현재 이슈를 먼저 끝낸다.

### 3.2 다른 팀원에게 영향 가는 변경을 독단적으로 하지 않는다

아래는 먼저 합의 없이 바꾸지 않는다.

- 공통 계약
- 공통 타입
- 공통 store
- 공통 API 응답 구조
- 공통 route path

### 3.3 설계 문서에 맞게 구현한다

- 구현 편의를 이유로 설계를 우회하지 않는다.
- 설계를 바꿔야 한다면 먼저 설계 문서를 수정하고, 그 다음 구현한다.
- "일단 짜고 나중에 문서 맞추자"는 금지한다.

### 3.4 컨벤션을 계속 확인한다

작업 중 아래를 계속 점검한다.

- 슬라이스 경계
- import 규칙
- 타입 import 규칙
- page / section 역할
- store 위치

### 3.5 중간 상태도 항상 살아 있어야 한다

각 단계가 끝날 때마다 최소 아래는 통과해야 한다.

```bash
pnpm run lint
pnpm run tsc
```

### 3.6 unrelated 변경을 끼워 넣지 않는다

작업 중 자주 나오는 위험 신호:

- "이것도 같이 고치면 좋겠다"
- "이 파일도 건드리면 더 예쁘다"
- "이참에 구조도 조금 바꾸자"

현재 이슈 달성에 직접 필요하지 않으면 하지 않는다.

## 4. 구현 후 반드시 할 것

### 4.1 설계 문서와 다시 비교

구현이 끝나면 먼저 작업자가 직접 확인한다.

- 설계대로 구현됐는가?
- 설계와 다른 부분이 있다면 문서가 먼저 바뀌었는가?

### 4.2 객관적인 시선으로 한 번 더 검토

구현자 시선이 아니라 리뷰어 시선으로 다시 본다.

확인 항목:

- 책임 분리가 맞는가?
- 범위 초과 변경이 없는가?
- 공통 계약 위반이 없는가?
- 다른 팀원의 작업에 영향이 가지 않는가?

### 4.3 관련 없는 변경 제거

최종 diff를 보고 현재 이슈와 무관한 파일 수정은 제거한다.

### 4.4 커밋 전 검증

모든 커밋 전에 반드시 수행한다.

```bash
pnpm run lint
pnpm run tsc
```

### 4.5 PR 전 검증

PR 전에는 반드시 수행한다.

```bash
pnpm run build
```

### 4.6 최종 보고 형식

최종 보고에는 최소 아래 세 가지를 포함한다.

- 무엇을 바꿨는지
- 설계와 어떻게 맞췄는지
- 남은 리스크가 무엇인지

## 5. 즉시 중단하고 재정렬해야 하는 상황

아래 상황에서는 구현을 멈추고 먼저 재정렬한다.

- 현재 이슈와 다른 주제로 작업이 퍼지기 시작할 때
- 다른 팀원의 소유 파일을 넓게 건드리려 할 때
- 공통 계약을 바꾸고 있는데 사전 합의가 없을 때
- 설계 문서와 구현 방향이 충돌하는데 무시하고 진행하려 할 때
- 임시방편을 핵심 구조에 넣으려 할 때

대응 원칙:

- 구현을 멈춘다
- 범위를 다시 정의한다
- 필요하면 설계 문서를 먼저 수정한다

---

## 6. Codex 사용 기본 순서

Codex를 사용할 때는 아래 순서를 기본값으로 한다.

1. 현재 구현 분석
2. 설계 문서 확인 또는 설계 작성
3. 작은 단계로 구현 계획 수립
4. 단계별 구현
5. 단계별 `lint`, `tsc` 검증
6. 설계 대비 자기 검토
7. 객관적 재검토
8. 커밋

즉, **바로 코드를 쓰는 것이 아니라 `분석 -> 설계 -> 작은 단계 구현 -> 재검토` 순서로 움직인다.**

---

## 7. 금지 사항

- 현재 이슈와 무관한 리팩토링 추가
- 다른 팀원 소유 범위까지 넓게 수정
- 설계 문서 없이 구조를 새로 뒤집는 작업
- 공통 계약을 독단적으로 변경하는 작업
- 검증 없이 커밋하는 작업

---

## 요약

AI 협업의 핵심은 속도가 아니라 **일관성**이다.

이번 프로젝트에서 Codex는 아래 순서를 어기지 않는다.

**분석 -> 설계 -> 작은 단계 구현 -> 검증 -> 설계 대비 검토 -> 객관적 재검토**
