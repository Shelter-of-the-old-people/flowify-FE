# 메일 요약 단일 결과 집계 FE 수정안

> **작성일:** 2026-05-04  
> **대상:** Frontend  
> **관련 이슈:** `메일 요약 후 전달` 템플릿 3종 고도화  
> **관련 화면:** `/templates`, `/templates/:id`, 워크플로우 에디터 sink 설정 패널

---

## 1. 개요

`메일 요약 후 전달` 템플릿 3종은 최종 결과를 Slack 메시지 1개 또는 Notion 페이지 1개로 유지하는 방향이 사용자 경험상 가장 자연스럽다.

다만 현재는 메일 여러 개를 한 번에 자유 요약하는 구조라서:

- 일부 메일이 결과에서 빠질 수 있고
- 템플릿 설명과 실제 동작이 다르게 보이며
- Slack, Notion 설정도 ID 직접 입력 중심이라 사용성이 낮다.

이 문서는 FE가 이 구조를 사용자에게 어떻게 설명하고, 어떤 방식으로 설정 UX를 개선해야 하는지 정리한다.

---

## 2. 목표 UX

목표는 아래 한 줄로 정리할 수 있다.

`메일별 정보는 유지하되, 최종 결과는 1개로 보이게 한다.`

사용자가 보게 될 결과 기대치는 다음과 같다.

- Slack: 메일별 항목이 구분된 메시지 1개
- Notion: 메일별 항목이 구분된 페이지 1개
- 템플릿 설명: 메일을 하나씩 보내는 것이 아니라 메일 목록을 정리한 단일 결과를 만든다고 안내

---

## 3. 현재 FE 문제

### 3.1 템플릿 설명과 실제 동작 불일치

현재 일부 템플릿 이름과 설명은 `메일을 하나씩 요약해서 보낸다`는 인상을 준다.

하지만 실제 런타임은:

- 읽어온 메일 목록을 모아서
- 결과 1개를 만들고
- Slack 또는 Notion으로 한 번 전달하는 흐름에 가깝다.

### 3.2 Slack / Notion 설정 UX 불편

현재 sink 설정은 다음과 같은 문제가 있다.

- Slack 채널 ID 직접 입력 필요
- Notion 페이지 ID 직접 입력 필요
- 선택 후 사람이 읽는 이름보다 raw ID가 먼저 보임

이 구조는 테스트 단계에서도 실수가 잦고, 일반 사용자에게는 진입 장벽이 높다.

### 3.3 현재 동작 기준 안내 부족

템플릿 상세에서 아래 내용이 명확히 드러나지 않는다.

- 결과가 1개라는 점
- 메일별 정보가 어떤 형식으로 묶여 나가는지
- Slack/Notion이 현재는 ID 입력 기반이라는 점

---

## 4. FE 수정 목표

### 4.1 템플릿 설명 보정

메일 템플릿 3종은 현재 런타임 기준으로 설명을 보정해야 한다.

권장 문구:

- `읽지 않은 메일 목록 요약 후 Slack 공유`
- `중요 메일 목록 요약 후 Notion 저장`
- `중요 메일 목록에서 할 일 추출 후 Notion 저장`

설명도 아래처럼 맞춘다.

- 메일을 하나씩 전송하는 템플릿이 아니라
- 메일 목록을 정해진 형식으로 정리해
- 최종 결과 1개로 공유/저장하는 템플릿이라고 안내한다.

### 4.2 결과 형식 안내 추가

템플릿 상세에 `현재 동작 기준` 안내 영역을 유지하거나 강화한다.

권장 안내 포인트:

- 여러 메일을 하나의 결과로 정리한다
- 결과 안에서는 메일별 항목이 구분된다
- Slack은 메시지 1개, Notion은 페이지 1개를 목표로 한다

### 4.3 Slack / Notion picker UX 추가

FE에서는 Google Drive 폴더 picker와 같은 `RemoteOptionPicker` 패턴을 재사용해 Slack / Notion sink picker를 연결한다.

1차 구현 기준으로 제공하는 picker UX는 다음과 같다.

- Slack `channel_picker`
  - 기존 채널 목록 선택
  - 채널 이름 표시
  - 선택값은 ID 저장
- Notion `page_picker`
  - 접근 가능한 페이지 목록 선택
  - 페이지 제목 표시
  - 선택값은 ID 저장

1차 범위에서 공통으로 지키는 원칙:

- 기존 항목 선택까지만 지원
- 설정 패널 안에서 새 채널이나 새 부모 페이지를 만드는 UX는 포함하지 않음

생성 관련 범위는 다음과 같이 구분한다.

- Slack
  - 1차: 기존 채널 선택
  - 2차: 새 채널 생성 검토
- Notion
  - 1차: 기존 부모 페이지 선택
  - 2차: 새 부모 페이지 생성 검토
  - 비고: 실행 결과 페이지를 선택한 부모 페이지 아래에 새로 만드는 동작과, 설정 패널에서 부모 페이지 자체를 새로 만드는 UX는 구분해서 본다.

### 4.4 결과 표시 방식 통일

템플릿 목록과 상세에서 메일 템플릿은 아래 공통 인상을 줘야 한다.

- `메일별 구조를 유지한 단일 결과`
- `즉시 공유/저장에 적합한 정리형 템플릿`

## 5. 세부 수정안

### 5.1 템플릿 목록

수정 포인트:

- 카테고리 배지 유지
- 메일 템플릿은 보정된 description 사용
- 필요 서비스 표기는 사람이 읽기 쉬운 이름으로 유지
- 필요하면 `결과 1개` 성격이 드러나는 짧은 메타 문구 추가 검토

예:

- `메일 요약 후 전달 · 필요 서비스 2개`

### 5.2 템플릿 상세

수정 포인트:

- 제목 아래 설명을 현재 런타임 기준 문구로 보정
- `현재 동작 기준` 박스 유지
- 결과 예시를 보여줄 수 있으면 가장 좋음

예시 출력 형태:

```text
읽지 않은 메일 요약 5건

1. 발신자: Slack
- 제목: 확인 코드
- 핵심: 이메일 확인 코드 안내
- 액션: 코드 입력 필요
```

### 5.3 워크플로우 에디터 sink 설정

수정 포인트:

- Slack: 채널 ID 직접 입력 대신 채널 선택
- Notion: 페이지 ID 직접 입력 대신 페이지 선택
- 선택 후에는 `선택된 채널`, `선택된 페이지` 식으로 사람이 읽는 라벨 표시
- raw ID는 보조 정보로만 남김
- Google Drive `folder_picker`와 동일한 패널 흐름을 재사용하되, Slack / Notion은 경로 탐색 없이 검색 + 선택 중심으로 동작

### 5.4 안내 문구

Slack 안내:

- 앱이 채널에 초대되지 않았으면 전송이 실패할 수 있습니다.

Notion 안내:

- 공유된 페이지 또는 접근 가능한 페이지만 목록에 표시됩니다.
- 새 결과는 선택한 부모 페이지 아래에 생성됩니다.

## 6. FE 범위와 비범위

### 6.1 FE에서 바로 가능한 것

- 설명 문구 보정
- 결과 형식 안내
- picker UI 연결
- 선택값 라벨 표시
- 설정 패널 레이아웃 개선

### 6.2 백엔드 지원이 필요한 것

- Slack 채널 목록 API
- Notion 페이지 목록 API
- Gmail fetch 개수 설정화
- 실제 메일별 구조화 후 단일 결과 집계 런타임

즉 FE는 화면과 사용 흐름을 개선할 수 있고, 현재는 Slack / Notion picker UI까지 연결된 상태다. 다만 picker 데이터와 실행 품질은 백엔드 지원이 있어야 완성된다.

## 8. 결론

FE 관점에서 이번 전략의 핵심은 다음과 같다.

- 메일을 하나씩 따로 보내는 템플릿처럼 보이지 않게 설명한다.
- 최종 결과는 1개지만, 메일별 정보가 보존된다는 기대를 명확히 보여준다.
- Slack 채널, Notion 페이지 선택을 ID 입력이 아닌 목록 선택으로 바꿔 사용성을 크게 낮춘다.

이 방향으로 가야 `메일 요약 후 전달` 템플릿 3종이 실제 사용자에게도 자연스럽게 보인다.
