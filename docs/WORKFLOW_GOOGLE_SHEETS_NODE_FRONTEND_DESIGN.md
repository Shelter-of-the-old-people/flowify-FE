# Workflow Google Sheets Node Frontend Design

> 작성일: 2026-05-11
> 대상 화면: `/workflows/:id`
> 범위: flowify-FE Google Sheets UX 및 에디터 동작
> 관련 저장소: `flowify-BE-spring`, `flowify-BE`

---

## 1. 목적

이 문서는 Google Sheets를 워크플로우의 정식 서비스로 다루기 위한 프론트엔드 설계를 정리한다.

Google Sheets는 아래 세 역할을 모두 지원해야 한다.

- 시작 노드
- 중간 노드
- 끝 노드

프론트엔드의 목표는 단순히 기능을 노출하는 것이 아니라, 일반 사용자가 설정 흐름을 보고 바로 이해할 수 있게 만드는 것이다.

---

## 2. 제품 목표

Google Sheets 경험은 아래와 같은 대표 자동화를 자연스럽게 지원해야 한다.

- 시트 전체를 읽어 요약하거나 정리하기
- 새 행이 들어오면 후속 자동화 시작하기
- 특정 행이 수정되면 다시 처리하기
- 특정 단어가 들어간 행 찾기
- 기준 컬럼으로 한 행 찾기
- 결과를 시트에 누적 저장하기
- 보고서 범위를 덮어쓰기
- 기준 컬럼으로 기존 행 수정 또는 업서트하기
- 원하는 스프레드시트 파일이 없으면 바로 만들기
- 원하는 시트 탭이 없으면 바로 만들기

## 2.1 현재 사용자 흐름 제한

현재 FE 사용자 흐름에서는 `다음 단계 -> 중간 처리 추가`가 항상 공통 `data-process` 노드만 생성한다.

따라서 아래 Google Sheets 중간 기능은 코드와 패널은 구현되어 있어도, 현재 사용자가 정상 경로로 직접 추가할 수는 없다.

- `read_range`
- `search_text`
- `lookup_row_by_key`

즉 현재 사용자 기준으로 바로 사용할 수 있는 Google Sheets 경로는 아래 두 가지다.

- 시작 노드로 Google Sheets를 읽기
- 끝 노드로 Google Sheets에 저장하기

중간 Google Sheets 노드 설계 내용은 이후 사용자 생성 경로가 열릴 때 적용 대상이며, 현재 화면에서는 노출하지 않는다.

---

## 3. 주요 사용자 시나리오

### 3.1 전체 읽기와 정리

- 시트 전체를 읽어 요약한다.
- 특정 범위만 읽어 보고서를 만든다.
- 읽은 표 데이터를 다음 노드로 전달한다.

### 3.2 검색과 분류

- 특정 단어가 포함된 행만 찾는다.
- 특정 컬럼에서 정확히 일치하는 값을 찾는다.
- 찾은 결과를 다른 시트에 저장한다.

### 3.3 대기열과 상태판

- 새 행이 추가되면 자동화를 시작한다.
- 특정 행이 수정되면 다시 처리한다.
- 처리 결과를 상태 컬럼으로 되돌려 적는다.

### 3.4 기준표 조회

- 이메일, 코드, 학생 번호 같은 키로 한 행을 찾는다.
- 찾은 행을 정책표나 참조 데이터로 사용한다.

### 3.5 결과 저장

- 기존 데이터 아래에 새 행을 추가한다.
- 특정 범위를 새 결과로 덮어쓴다.
- 같은 키를 가진 행이 있으면 수정하고, 없으면 추가한다.

### 3.6 대상이 없을 때 바로 생성

- 원하는 스프레드시트 파일이 목록에 없으면 즉시 생성한다.
- 파일을 선택한 뒤 원하는 시트 탭이 없으면 즉시 생성한다.

## 4. 지원 노드 역할

### 4.1 시작 노드

지원 모드:

- `sheet_all`
- `new_row`
- `row_updated`

필수 입력:

- 스프레드시트 파일
- 시트 탭
- 선택 범위
- 헤더 행 번호
- 데이터 시작 행 번호

모드별 추가 입력:

- `new_row`
  - `initial_sync_mode`
- `row_updated`
  - `initial_sync_mode`
  - `key_column`

### 4.2 중간 노드

지원 액션:

- `read_range`
- `search_text`
- `lookup_row_by_key`

필수 입력:

- 스프레드시트 파일
- 시트 탭
- 선택 범위

액션별 추가 입력:

- `search_text`
  - 검색 소스
  - 검색 값 또는 입력 필드 바인딩
  - 검색 대상 컬럼
  - 검색 방식
  - 대소문자 구분
  - 결과 제한 수
- `lookup_row_by_key`
  - `key_column`
  - 조회 소스
  - 조회 값 또는 입력 필드 바인딩

### 4.3 끝 노드

지원 저장 방식:

- `append_rows`
- `overwrite_range`
- `update_row_by_key`
- `upsert_row_by_key`

필수 입력:

- 스프레드시트 파일
- 시트 탭

모드별 추가 입력:

- `overwrite_range`
  - `range_a1`
- `update_row_by_key`
  - `key_column`
- `upsert_row_by_key`
  - `key_column`

## 5. 선택 흐름

Google Sheets 선택 흐름은 두 단계로 구성한다.

### 5.1 스프레드시트 선택

사용자는 먼저 스프레드시트 파일을 선택한다.

지원 동작:

- 기존 스프레드시트 파일 탐색
- 스프레드시트 파일 검색
- 원하는 파일이 없으면 새 스프레드시트 생성

생성 규칙:

- 새 스프레드시트는 picker 안에서 즉시 만든다.
- 생성 성공 후 방금 만든 파일을 자동 선택 상태로 만든다.

### 5.2 시트 탭 선택

스프레드시트가 선택된 뒤, 그 안의 시트 탭을 고른다.

지원 동작:

- 기존 시트 탭 탐색
- 원하는 탭이 없으면 새 시트 생성

생성 규칙:

- 같은 이름의 시트가 이미 있으면 기존 탭을 재사용한다.
- 없으면 새로 만들고 즉시 선택한다.

---

## 6. 선택 ID 규칙

같은 스프레드시트 안에는 여러 시트 탭이 있으므로, 시트 탭 선택 항목의 ID를 단순 `spreadsheetId`로 쓰면 안 된다.

프론트엔드 규칙:

- 스프레드시트 파일의 실 ID는 `spreadsheetId`를 사용한다.
- 시트 탭 선택 UI는 고유한 선택용 ID를 사용한다.

현재 시트 선택용 ID 형식:

- `spreadsheetId::sheet::sheetName`

중요 규칙:

- 이 값은 UI 선택용이다.
- 실제 저장값은 계속 아래를 사용한다.
  - `spreadsheet_id`
  - `sheet_name`

즉, 프론트엔드는 UI에서 쓰는 고유 ID를 실제 워크플로우 대상 ID로 저장하면 안 된다.

## 7. 저장 모델

### 7.1 시작 노드

Google Sheets 시작 노드는 아래 값을 저장해야 한다.

- `target = spreadsheet_id`
- `target_label = spreadsheet title`
- `sheet_name`
- `header_row`
- `data_start_row`
- `initial_sync_mode`
- 필요 시 `key_column`

### 7.2 중간 노드

Google Sheets 중간 노드는 단순 문자열 target이 아니라 구조화된 액션 설정을 저장해야 한다.

### 7.3 끝 노드

Google Sheets 끝 노드는 아래 값을 저장해야 한다.

- `spreadsheet_id`
- `sheet_name`
- `write_mode`
- 필요 시 `range_a1`
- 필요 시 `key_column`

## 8. UX 요구사항

### 8.1 사람이 이해할 수 있는 저장 방식 문구

`append_rows`, `upsert_row_by_key` 같은 내부 이름만으로는 부족하다.

권장 문구:

- `append_rows`
  - 기존 내용 아래에 새 행 추가
- `overwrite_range`
  - 선택한 범위를 새 결과로 덮어쓰기
- `update_row_by_key`
  - 같은 기준값을 가진 기존 행만 수정
- `upsert_row_by_key`
  - 같은 행이 있으면 수정하고, 없으면 새로 추가

### 8.2 저장 위치 설명

범위 입력 규칙:

- 사용자가 `A1`, `A1:B10`처럼 시트 이름 없는 범위를 입력하면, 선택한 시트 탭 기준 범위로 해석한다.
- 사용자가 이미 `MailSubset!A1:B10`처럼 시트 이름까지 넣으면 그 값을 그대로 사용한다.
- 화면 설명도 `선택한 시트 탭 기준 범위`라는 점을 드러내야 한다.

사용자는 아래 세 가지를 명확히 이해해야 한다.

- 어떤 스프레드시트 파일에 저장되는지
- 어떤 시트 탭에 저장되는지
- 필요한 경우 어떤 범위를 쓰는지

현재 기준:

- 중간 노드와 끝 노드 모두 선택된 스프레드시트 파일과 시트 탭을 분리해서 보여준다.
- 사용자가 `Sheet2`를 선택했는데 `Sheet1`처럼 보이는 혼란이 없어야 한다.

### 8.3 기준 컬럼 설명

`update`, `upsert`, `row_updated`, `lookup`에서는 `key_column`이 핵심이다.

예시:

- `email`
- `order_id`
- `student_id`

### 8.4 첫 실행 설명

`initial_sync_mode`는 내부 이름만 보여주지 말고 의미를 함께 설명해야 한다.

권장 문구:

- `skip_existing`
  - 현재 행은 무시하고 이후에 들어오는 변경부터 처리
- `emit_existing`
  - 첫 실행에서 기존 행도 함께 처리

### 8.5 표 형태 미리 이해시키기

사용자는 어떤 컬럼 구조로 시트에 들어가는지 미리 이해할 수 있어야 한다.

현재 기준:

- 저장 방식은 버튼 이름만이 아니라 설명 카드 형태로 보여준다.
- `append`, `overwrite`, `update`, `upsert`가 각각 시트에 어떻게 반영되는지 바로 읽을 수 있어야 한다.
- 시작 노드의 `new_row`, `row_updated`는 첫 실행 기준을 설명 카드 형태로 보여준다.

다음 UX 보완 단계에서 고려할 항목:

- 샘플 행 미리보기
- 출력 컬럼 미리보기
- 헤더 포함 여부 설명 강화

## 9. 현재 프론트엔드 상태

구현됨:

- Google Sheets 시작, 중간, 끝 노드 설정
- 스프레드시트 picker
- 시트 탭 picker
- picker 안에서 새 스프레드시트 생성
- picker 안에서 새 시트 생성
- 시트 탭 선택 충돌을 막는 고유 선택 ID 적용
- 실제 저장값은 `spreadsheet_id`와 `sheet_name`으로 유지
- 시작 노드 첫 실행 기준 설명 카드
- 중간 노드 검색/조회 설명 보강
- 끝 노드 저장 방식 설명 카드
- 끝 노드와 중간 노드의 스프레드시트/시트 분리 표시

남은 UX 보완점:

- 표 형태 미리보기가 약하다.
- 저장 결과가 시트에 어떻게 들어가는지 설명이 더 필요하다.
- 실제 샘플 행 미리보기는 아직 없다.
- 헤더 포함 여부와 컬럼 매핑 설명은 더 강화할 수 있다.

## 10. 검증 기준

프론트엔드 동작이 올바르다고 볼 수 있는 조건:

- `Sheet2`를 선택했을 때 `Sheet1`로 되돌아가지 않는다.
- 새 스프레드시트를 만들면 즉시 그 파일이 선택된다.
- 새 시트를 만들면 즉시 그 탭이 선택된다.
- 저장된 워크플로우에는 UI 선택용 시트 ID가 아니라 실제 `spreadsheet_id`가 저장된다.
- 새로고침 후 시작 노드가 올바른 파일과 시트 탭을 복원한다.
- 중간 노드가 검색 및 조회 설정을 복원한다.
- 끝 노드가 올바른 저장 대상과 저장 방식을 복원한다.

## 11.1 공통 표 가공 경로

Google Sheets 경험은 `Gmail -> Sheets` 한 가지 흐름으로만 설계하면 안 된다.
프론트엔드는 시작 데이터가 무엇이든 중간에서 표 형태로 정리한 뒤 시트에 저장할 수 있는 흐름을 보여줘야 한다.

대표 시작 타입:

- `SINGLE_EMAIL`
- `SINGLE_FILE`
- `SPREADSHEET_DATA`

대표 표 가공 액션:

- `filter_fields_table`
  - 여러 입력 타입에서 공통으로 지원한다.
  - 선택한 필드만 골라 시트 컬럼으로 정리한다.
- `filter_metadata_table`
  - 파일 메타데이터를 시트 컬럼으로 정리한다.

대표 사용자 시나리오:

- Gmail 메일을 `메일 ID / 제목 / 발신자 / 날짜 / 본문 미리보기` 표로 정리해서 시트에 저장
- Google Drive 파일 메타데이터를 `파일명 / 링크 / 업로드 시간 / 크기` 표로 정리해서 시트에 저장
- 기존 시트 데이터를 다시 골라 다른 시트용 표로 재정리

프론트엔드 UX 원칙:

- `표로 정리해서 저장` 같은 문구를 써서 사용자가 결과 형태를 바로 이해할 수 있게 한다.
- 내부 타입 이름인 `SPREADSHEET_DATA`는 화면 전면에 드러내지 않는다.
- 같은 `DATA_FILTER` 계열이라도 `텍스트로 정리`와 `표로 정리`를 명확히 구분해서 보여준다.
- 시트 저장용 가공 액션은 Gmail 전용처럼 보이지 않게 공통 패턴으로 노출한다.

## 11.2 향후 보완점

현재 에디터의 `다음 단계 -> 중간 처리 추가` 흐름은 공통 `data-process` 노드만 생성한다.

따라서 Google Sheets 중간 노드 기능인 아래 항목은 코드와 패널 구현은 존재하지만, 현재 사용자 흐름에서는 직접 추가하거나 설정할 수 없다.

- `read_range`
- `search_text`
- `lookup_row_by_key`

이번 이슈에서는 이 경로를 화면에서 가능한 기능처럼 노출하지 않고, 제한사항과 향후 보완점으로만 문서화한다.

향후 에디터가 공통 `data-process` 외의 중간 노드 타입을 직접 생성할 수 있게 되면, 그 시점에 Google Sheets 중간 노드 노출과 설정 UX를 다시 열어야 한다.

## 11. 결정 요약

프론트엔드의 책임은 아래와 같다.

- 사용자가 스프레드시트와 시트를 쉽게 선택하고 만들게 한다.
- 시트 탭 선택 충돌이 없도록 고유 ID를 유지한다.
- 실제 저장은 안정적인 구조값으로 저장한다.
- 일반 사용자가 저장 동작을 이해할 수 있도록 충분히 설명한다.

프론트엔드가 직접 책임지지 않는 영역:

- Google Sheets diff 계산
- durable state commit
- 실제 Google Sheets API 읽기/쓰기 실행

이 영역은 `flowify-BE`와 `flowify-BE-spring`이 담당한다.
